# SYSTEM_OVERVIEW

## Цель системы
Построить управляемую и практичную систему разработки, в которой:
- ты ставишь задачи в Telegram обычным языком;
- OpenClaw берет на себя роль оркестратора (уточняет, планирует, маршрутизирует);
- Claude Code выполняет реальные изменения в коде в твоей рабочей среде;
- качество контролируется через review/test шаги;
- затраты токенов и сложность не выходят из-под контроля.

Ключевая идея: **не заменять текущий рабочий процесс, а усилить его**.

---

## Почему мы выбрали эту архитектуру
Мы не идем в крайности:
- не делаем «только Claude Code вручную» (слишком много рутины руками);
- не делаем «полную автономию на сервере» сразу (слишком много рисков и сложности).

Архитектура v1 опирается на два подтвержденных подхода:
- **GSD (Get-shit-done) дисциплина**: фазы plan -> implement -> review -> tests + фиксация состояния в документации.
- **Context Mode дисциплина**: в модель отправляем выжимку и структурированный результат, а не сырые большие tool outputs.

Выбран компромисс:
1. сохранить сильные стороны твоего текущего Claude Code;
2. добавить оркестрацию, контекст и автоматизацию через OpenClaw;
3. внедрять поэтапно, с проверкой пользы на каждом шаге.

---

## Архитектура v1 (как работает система)

### 1) OpenClaw (оркестратор, интерфейс в Telegram)
Что делает:
- принимает тикет/задачу;
- задает уточняющие вопросы;
- формирует структурированный план (scope, шаги, критерии готовности);
- решает, какой режим нужен дальше (`implement`, `review`, `tests`);
- отслеживает статус выполнения и дает тебе понятный апдейт.

Что не делает:
- не выполняет опасные автономные действия без согласования;
- не пушит напрямую в `main`.

### 2) Bridge backend (тонкая интеграционная прослойка)
Что делает:
- принимает задачу от OpenClaw в стандартизированном формате;
- запускает соответствующий режим Claude Code;
- возвращает сжатый результат (summary) + ссылки/идентификаторы артефактов;
- обеспечивает «контекстную экономию» (не льет сырые логи в LLM).

Зачем нужен:
- разделяет оркестрацию и исполнение;
- дает контролируемый контракт между системами;
- позволяет масштабировать процесс без хаоса.

### 3) Claude Code (локально у тебя)
Что делает:
- вносит изменения в код в твоих репозиториях;
- работает в изолированной ветке (`agent/*`);
- запускает проверки/тесты;
- готовит PR.

Почему локально:
- у тебя уже есть рабочая среда и настройки;
- меньше риска рассинхронизации между сервером и твоим дев-окружением;
- проще и надежнее для MVP.

---

## Идеальный рабочий сценарий (target flow)
1. Ты отправляешь тикет в Telegram.
2. OpenClaw формирует и согласует с тобой план.
3. После подтверждения задача уходит через bridge в Claude Code.
4. Claude Code делает изменения в ветке и запускает проверки.
5. Включается review/test контур (при необходимости с фикс-итерациями).
6. На выходе: готовый PR + краткий отчет, что сделано и что проверить вручную.

Результат: ты управляешь процессом на уровне решений, а рутину исполнения и контроля система берет на себя.

---

## Ограничения и границы v1
- Не строим «зоопарк» агентов на старте.
- Не подключаем серые схемы токенов/авторизации.
- Не автоматизируем всё сразу — сначала проверяем, что MVP реально ускоряет работу.

---

## Критерии успеха
Система считается удачной, если через короткий цикл использования (например, 7 дней):
- уменьшает ручную рутину минимум на заметную величину;
- не раздувает расход токенов до неадекватного уровня;
- дает стабильный путь от тикета до PR;
- не ломает привычный рабочий процесс.

---

## Принципы, которые не меняем
- Human-in-the-loop на ключевых решениях.
- PR-only: никаких прямых изменений в `main`.
- Сначала понятность и надежность, потом усложнение.
- «Меньше магии — больше предсказуемости».
