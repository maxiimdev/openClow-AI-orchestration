# TOKEN_POLICY

## Цель
Сделать оркестрацию экономной и предсказуемой по расходу токенов, без потери качества результата.

Ключевая установка: **не платить токенами за лишнюю сложность процесса**.

---

## Основные принципы

### 1) Cheap mode по умолчанию
Для типовых задач используем компактный контекст и короткие ответы.

### 2) Deep mode только вручную
Глубокий режим включается явно, когда действительно нужен (сложная архитектура, спорные кейсы, крупный refactor).

### 3) Один planner на старте
Не умножаем число промежуточных агентов без доказанной пользы.

### 4) Scope-control
Ограничиваем размер одной итерации (обычно 5–7 файлов).

### 5) Context compression
Сырые большие логи/выгрузки не отправляем в LLM как есть.

### 6) Structured payloads
Предпочитаем короткий JSON вместо длинной прозы.

---

## Практика контекстной экономии (Context Mode подход)
1. Инструменты работают в изоляции.
2. В LLM-контекст попадает только:
   - краткая выжимка;
   - идентификатор/ссылка на полный артефакт.
3. Полный артефакт читается только при необходимости и точечно.

Плюс: резко снижается «контекстный шум» и стоимость итераций.

---

## Операционные лимиты v1
- review/patch loop: максимум 2–3 итерации;
- при росте scope — декомпозиция;
- запрет на бесконечные цепочки вызовов без решения.

---

## Что мониторим регулярно
1. Tokens in/out по сессии.
2. Долю кэша (cache hit).
3. Среднее число итераций на задачу.
4. Время от `new` до `ready_pr`.
5. Долю задач, где deep mode реально понадобился.

Цель: видеть, где система экономит, а где начинает «жечь» бюджет.

---

## Триггеры оптимизации
Если замечаем любой из признаков:
- много длинных ответов без реальной пользы;
- частые повторы одинакового контекста;
- задачи часто уходят в >3 итераций;
- токены растут быстрее, чем качество;

то делаем одно из действий:
1. сжимаем payload/план;
2. режем scope итерации;
3. убираем лишний промежуточный шаг;
4. переводим часть информации в постоянные docs вместо повтора в промпте.

---

## Правило приоритета
1) качество результата
2) стабильность процесса
3) экономия токенов

Экономим токены агрессивно, но не ценой сломанной разработки.
