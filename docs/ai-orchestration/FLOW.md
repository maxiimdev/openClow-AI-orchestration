# FLOW

## Зачем нужен этот flow
Этот пайплайн нужен, чтобы каждая задача проходила одинаково предсказуемый путь:
- без хаотичных «сразу писать код»;
- с контролем качества;
- с ограничением итераций и токенов;
- с понятной точкой ответственности на каждом шаге.

Этот flow напрямую отражает практику GSD: обсуждение/планирование -> выполнение -> проверка -> фиксация результата.

Идея: **процесс должен быть повторяемым**. Тогда система масштабируется и не ломается от роста задач.

---

## State machine (v1)
`new -> plan -> approved -> implement -> review -> patch (optional loop) -> tests -> ready_pr -> done`

Дополнительные служебные статусы:
- `blocked` — нужны данные/доступы/решение пользователя
- `escalated` — после лимита итераций или критической неопределенности
- `canceled` — задача отменена пользователем

---

## Подробно по этапам

### 1) `new`
**Вход:** тикет/идея/описание задачи от пользователя.

**Цель:** зафиксировать задачу в рабочем формате.

**Выход:** черновик задачи с ключевым контекстом.

---

### 2) `plan`
**Что делает оркестратор:**
- уточняет недостающие требования;
- формирует план в структурированном формате;
- ограничивает scope итерации (обычно до 5–7 файлов);
- прописывает acceptance criteria.

**Опционально (complex/deep plan):**
- подключает один planner sub-agent для предварительного варианта плана;
- проверяет/нормализует результат и только потом выносит план пользователю.

**Минимальная структура плана:**
- `summary`
- `scope`
- `change_plan`
- `acceptance_criteria`
- `risks`

**Выход:** план, готовый к подтверждению.

---

### 3) `approved`
**Что происходит:**
- пользователь подтверждает план или вносит правки;
- без подтверждения задача дальше не идет.

**Выход:** финализированный payload на implement.

---

### 4) `implement`
**Что делает bridge + Claude Code:**
- запускает режим `implement`;
- делает изменения в изолированной ветке `agent/*`;
- сохраняет краткий лог изменений.

**Выход:** набор изменений + список затронутых файлов + статус исполнения.

---

### 5) `review`
**Что делает режим review:**
- критически проверяет изменения;
- ищет дефекты, риски и нарушения правил;
- формирует замечания с приоритетами.

**Выход:**
- либо `pass` (можно дальше),
- либо переход в `patch`.

---

### 6) `patch` (опциональный цикл)
**Что делает implementer:**
- исправляет замечания review;
- возвращает задачу обратно в `review`.

**Ограничение:**
- цикл `review <-> patch` максимум 2–3 итерации (v1).
- если критика не закрывается — `escalated`.

---

### 7) `tests`
**Что делает test-режим:**
- запускает тесты/чеклист;
- фиксирует, что прошло/не прошло;
- выделяет блокеры.

**Выход:** тестовый статус для решения о PR.

---

### 8) `ready_pr`
**Условия входа:**
- implement завершен;
- review пройден или замечания закрыты;
- тестовый статус приемлемый.

**Что делаем:**
- формируем PR;
- прикладываем краткий итог: что сделано, риски, ручные проверки.

---

### 9) `done`
**Финал:**
- задача закрыта;
- зафиксированы ключевые итоги;
- при необходимости обновлены docs/decisions.

---

## Правила остановки и эскалации
1. После лимита review/patch итераций — эскалация пользователю.
2. При распухании scope — декомпозиция на новые микро-задачи.
3. При нехватке данных/доступов — статус `blocked`.
4. При сомнительных/рискованных действиях — обязательное подтверждение пользователя.

---

## Принципы выполнения
- Идем маленькими, контролируемыми итерациями.
- Не жертвуем качеством ради скорости.
- Не жертвуем токенами ради «красоты процесса».
- Planner sub-agent используем условно (по триггерам), а не постоянно.
- Любой шаг должен быть объясним и воспроизводим.
