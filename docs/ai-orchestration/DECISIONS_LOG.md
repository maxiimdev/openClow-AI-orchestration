# DECISIONS_LOG

## Назначение
Этот файл фиксирует ключевые архитектурные и процессные решения.

Зачем:
- не терять контекст договоренностей;
- понимать, почему выбрали именно такой путь;
- избегать повторных споров по уже решенным вопросам;
- быстро вводить нового участника/агента в курс дела.

Формат записи: **дата -> решение -> причина -> последствия**.

---

## 2026-03-02 — Baseline decisions (v1)

### D-001: Стратегия инструментов
**Решение:**
Используем связку OpenClaw + Claude Code, а не модель «только один инструмент».

**Причина:**
OpenClaw усиливает оркестрацию и контекст, Claude Code остается сильным исполнителем кода.

**Последствия:**
Гибкий процесс без потери привычного дев-флоу.

---

### D-002: Разделение ролей
**Решение:**
- OpenClaw = оркестратор
- Claude Code = исполнительный контур (`implement/review/tests`)

**Причина:**
Четкие границы ответственности уменьшают хаос.

**Последствия:**
Проще отладка, проще масштабирование.

---

### D-003: Размещение Claude Code
**Решение:**
На старте используем локальный Claude Code пользователя.

**Причина:**
У пользователя уже настроенная рабочая среда, навыки и стабильный процесс.

**Последствия:**
Минимум дублирования инфраструктуры, меньше операционного риска.

---

### D-004: Безопасность и ToS
**Решение:**
Не использовать извлеченные OAuth/Max токены Claude в OpenClaw.

**Причина:**
Юридические и операционные риски; придерживаемся безопасной модели интеграции.

**Последствия:**
Снижен риск блокировок и конфликтов с правилами платформ.

---

### D-005: Git-политика
**Решение:**
Работа только через отдельные ветки (`agent/*`) и PR-only путь.

**Причина:**
Нужна изоляция изменений и контроль качества перед merge.

**Последствия:**
Стабильность `main`, прозрачная история изменений.

---

### D-006: Оркестрация как state machine
**Решение:**
Использовать явные статусы задачи и ограниченный review/patch loop.

**Причина:**
Статусная модель делает процесс прогнозируемым и управляемым.

**Последствия:**
Меньше «подвисших» задач и бесконечных циклов.

---

### D-007: Экспертные роли через Expert Packs
**Решение:**
Экспертность внедряется через структурированные пакеты ролей, а не хаотичные длинные промпты.

**Причина:**
Нужна переносимая, проверяемая и обновляемая модель экспертности.

**Последствия:**
Рост качества без неконтролируемого усложнения.

---

### D-008: Экономика токенов
**Решение:**
Cheap mode по умолчанию, deep mode по явному запросу.

**Причина:**
Сдерживание расходов без потери качества на типовых задачах.

**Последствия:**
Контролируемый бюджет и предсказуемая стоимость итераций.

---

### D-009: Контекстная компрессия обязательна
**Решение:**
Большие tool outputs передавать как summary + artifact pointer.

**Причина:**
Сырые логи перегружают контекст и ухудшают эффективность.

**Последствия:**
Снижение контекстного шума и стоимости обработки.

---

### D-010: Документация как операционная база
**Решение:**
Вести архитектурную и процессную документацию в markdown с последующим переносом в Obsidian.

**Причина:**
Нужен единый источник правды для тебя, меня и будущих агентов.

**Последствия:**
Быстрый онбординг, меньше повторений, выше устойчивость системы.

---

### D-011: Берем принципы GSD, но без тяжелого фреймворка
**Решение:**
Использовать дисциплину GSD (фазность, атомарные шаги, фиксация состояния), но не тащить весь внешний комбайн команд/обвязки на старте.

**Причина:**
Нужен максимальный практический эффект при минимальной сложности MVP.

**Последствия:**
Процесс структурный и контролируемый, без лишней инфраструктуры.

---

### D-012: Context Mode как обязательная норма токен-экономии
**Решение:**
Считать контекстную компрессию и artifact-pointer модель обязательными для всех тяжелых tool outputs.

**Причина:**
Это напрямую снижает стоимость и контекстный шум без потери управляемости.

**Последствия:**
Стабильнее бюджет токенов и выше эффективность оркестрации.

---

### D-013: Берем анатомию промпта как рабочий стандарт v1
**Решение:**
Использовать структуру Task -> Context Files -> Brief -> Rules -> Clarification -> Plan -> Alignment как базовый протокол постановки задач Claude Code.

**Причина:**
Структурированная постановка задачи повышает качество выполнения и снижает риск ложных предположений модели.

**Последствия:**
Более предсказуемые планы, меньше случайных отклонений от требований.

---

### D-014: Токен-политика без крайностей
**Решение:**
Отказ от "экономии любой ценой" в пользу balanced-режима с контролируемыми диапазонами затрат.

**Причина:**
Цель — качество и стабильность, а не минимальные токены любой ценой.

**Последствия:**
Сохраняем качество на сложных задачах и избегаем перерасхода на рутине.

---

## 2026-03-04 — Baseline decisions (v2 update)

### D-015: Выделенный Project Expert Agent как постоянный «второй мозг»
**Решение:**
Добавить операционную модель с отдельным expert-agent контуром (spawned session), который хранит проектный контекст в MD-базе, готовит task packages и промпты для execution worker/Claude.

**Причина:**
Нужно уменьшить контекстный шум в main-оркестраторе и повысить стабильность качества постановки задач при росте сложности проекта.

**Последствия:**
- orchestration становится трехконтурной: main orchestrator -> project expert -> execution agent;
- повышается повторяемость промптинга и quality gate до исполнения;
- появляется обязательство дисциплинированно обновлять MD-документацию при новых решениях.
