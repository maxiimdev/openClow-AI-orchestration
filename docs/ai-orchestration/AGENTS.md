# AGENTS

## Зачем этот документ
Зафиксировать роли в системе так, чтобы:
- не было дублирования ответственности;
- каждый агент/режим понимал свою границу;
- процесс был масштабируемым без «зоопарка».

На v1 мы сознательно держим минимальный набор ролей.

---

## Роли v1

### 1) Orchestrator (OpenClaw)
**Роль:** управляющий контур процесса.

**Отвечает за:**
- intake задач и уточнение требований;
- формирование плана и согласование с пользователем;
- маршрутизацию в нужный режим (`implement/review/tests`);
- контроль статусов и коммуникацию;
- эскалацию при блокерах/рисках.

**Не отвечает за:**
- прямое редактирование production-кода без процесса;
- прямой push в `main`;
- бесконечные итерации без решения.

**Критерий качества роли:**
- процесс прозрачен, шаги понятны, лишней сложности нет.

---

### 2) Claude Implementer
**Роль:** исполнитель изменений в коде.

**Отвечает за:**
- реализацию утвержденного плана;
- соблюдение архитектурных и стилевых правил проекта;
- атомарные и объяснимые изменения;
- аккуратную работу в ветке `agent/*`.

**Не отвечает за:**
- изменение бизнес-требований «на ходу» без согласования;
- игнор review-замечаний;
- прямой merge в `main`.

**Критерий качества роли:**
- корректный код, соответствие scope, чистый diff.

---

### 3) Claude Reviewer
**Роль:** критический контроль качества и рисков.

**Отвечает за:**
- проверку корректности и поддерживаемости кода;
- поиск потенциальных дефектов и уязвимостей;
- внятные замечания с приоритетами;
- проверку соответствия проектным правилам.

**Не отвечает за:**
- переписывание задачи в новый scope;
- «вечный review» без практического выхода.

**Критерий качества роли:**
- замечания конкретные, воспроизводимые, полезные.

---

### 4) Claude Test Engineer
**Роль:** подтверждение работоспособности результата.

**Отвечает за:**
- запуск тестов/чеклистов;
- фиксацию блокирующих и неблокирующих проблем;
- формирование отчета по тестовому статусу.

**Не отвечает за:**
- изменение требований;
- попытку «задним числом» оправдать failing кейсы.

**Критерий качества роли:**
- ясный ответ: что прошло, что не прошло, что нужно проверить вручную.

---

## Как это реализуется на старте
На v1 эти роли чаще всего являются **режимами одного Claude Code** через bridge:
- `implement`
- `review`
- `tests`

Это снижает сложность и токен-расход, сохраняя ролевую дисциплину.

## Политика planner sub-agent (v1)
**Принцип:** planner sub-agent не включается always-on.

### По умолчанию
- планирование выполняет основной оркестратор (быстрый/дешевый путь).

### Когда включаем planner sub-agent
- задача крупная и неоднозначная;
- много ограничений/зависимостей;
- высокий архитектурный риск;
- пользователь явно запросил `deep plan`.

### Ограничения
- один planner sub-agent на задачу;
- одна итерация предварительного плана;
- итог всегда проходит финальную проверку основного оркестратора перед отправкой пользователю.

### Цель
Получить «второй мозг» там, где это реально повышает качество, без постоянного раздувания процесса и токен-расхода.

---

## Правила масштабирования ролей
Новые агенты добавляются только если есть подтвержденная потребность:
1. текущие роли регулярно перегружены;
2. появляется повторяемый отдельный тип задач;
3. есть измеримая польза от выделения роли.

Иначе — оставляем систему компактной.

## Эксперт-агент v1: Forum Prompt Architect
Добавлен отдельный узкий эксперт для этапа `plan`:
- готовит Task Package из сырого тикета;
- формирует senior-grade prompt для Claude Code;
- не исполняет код напрямую.

Ресурсы:
- `docs/ai-orchestration/experts/forum-prompt-architect/ROLE.md`
- `docs/ai-orchestration/experts/forum-prompt-architect/RULES.md`
- `docs/ai-orchestration/experts/forum-prompt-architect/CHECKLIST.md`
- `docs/ai-orchestration/experts/forum-prompt-architect/SOURCES.md`
- `docs/ai-orchestration/experts/forum-prompt-architect/PROMPT.md`

### Режим использования (компромисс)
- Для задач разработки по умолчанию: **architect-first** (сначала Forum Prompt Architect).
- Исключение: **trivial tasks** (очень простые/мелкие тикеты) — можно без эксперт-агента.
- Ручной override:
  - `full architect` — всегда через эксперт-агента
  - `skip architect` — выполнить без эксперт-агента
