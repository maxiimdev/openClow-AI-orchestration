# QA_GUARDRAILS (v1)

## Цель
Снизить риск критичных багов в любой фиче за счет обязательного QA-контура перед `ready_pr`.

Документ универсальный: не привязан к одному конкретному модулю.

---

## Где обычно максимальный риск (общие категории)
1. **Доменная логика** — сложные правила, состояния, переходы, роли, условия.
2. **Состояние и данные** — сохранение/сброс, кэш, стор, миграции состояния.
3. **Интеграции и контракты** — API-флаги, схемы, ошибки, разночтения front/back.
4. **Смежные сценарии** — когда правка в одном flow ломает другой.
5. **Негативные и edge-кейсы** — таймауты, ретраи, пустые/неполные данные, повторные действия.

> Примечание: в отдельных задачах риск-профиль может отличаться. Перед стартом отмечаем релевантные категории.

---

## Политика порядка работы (test strategy)
- По умолчанию: **test-first** для любой проверяемой бизнес-логики/состояний/контрактов.
- Исключение для frontend: если задача в основном про визуал/верстку (UI-first), это допустимо.
- Для UI-first обязательно:
  - заранее зафиксировать acceptance-критерии;
  - добавить минимум проверок на критичные сценарии (smoke/integration/e2e или эквивалент);
  - закрыть регрессионные риски до `ready_pr`.

## Обязательный preflight перед implement
- [ ] Границы задачи и сценарии использования описаны явно.
- [ ] Зафиксированы критичные инварианты (что не должно ломаться/теряться).
- [ ] Выделены 2–5 high-risk веток именно для этой задачи.
- [ ] Согласованы неоднозначности API/контрактов (или прописан fallback-план).
- [ ] Определен минимальный тестовый план: что проверяем обязательно.
- [ ] Явно выбран режим: `test-first` или `UI-first` (с обязательствами из политики выше).

---

## Обязательный review/test контур
Перед `ready_pr` должны быть закрыты все 4 блока (и 5-й для UI-задач):

### A) Contract checks (интеграционные)
- [ ] Проверены ключевые поля/флаги на целевых сценариях.
- [ ] Негативные ответы не ломают базовый пользовательский путь.
- [ ] Есть fallback для спорных/частично заполненных ответов.

### B) State integrity checks (целостность состояния)
- [ ] Критичные данные сохраняются и не теряются при переходах.
- [ ] Нет «протекания» состояния между разными сценариями.
- [ ] Повторный вход/перезапуск шага не ломает логику без явного правила.

### C) Flow regression checks (регресс)
- [ ] Базовый happy-path проходит.
- [ ] Критичные edge-path (минимум 2–3) проходят.
- [ ] Смежные сценарии, которые могли затронуться, не деградировали.

### D) Minimal automated tests
- [ ] Добавлен минимальный набор авто-тестов на ключевые инварианты.
- [ ] Для найденного бага добавлен regression test (где это практически возможно).

### E) UI verify (обязательно для UI-изменений)
- [ ] Проверены desktop + mobile viewport.
- [ ] Проверены ключевые UI states (default/hover/focus/disabled/error/loading).
- [ ] Приложены скриншоты и список отклонений expected vs actual.
- [ ] Проверки сделаны по `docs/ai-orchestration/UI_VERIFY_PROTOCOL.md`.

---

## Stop rules (нельзя переводить в `ready_pr`)
- Нет уверенности в сохранности критичных данных/состояний.
- Не проверены high-risk ветки, отмеченные на preflight.
- Смежный сценарий потенциально задет, но не проверен.
- Есть известный дефект без явного документированного компромисса.
- UI-задача не имеет UI evidence (скриншоты/отклонения/verdict).

---

## Практика «не убиться об рутину»
- Делить задачи на небольшие итерации.
- После каждого цикла фиксировать, что уже проверено (чтобы не проверять хаотично).
- Ручной регресс запускать по приоритетному short-list.
- Авто-тесты добавлять в точки с максимальной ценой ошибки.

---

## Обязательный deliverable в PR
1. Что изменено
2. Какие инварианты защищены
3. Какие high-risk сценарии проверены вручную
4. Какие тесты добавлены/обновлены
5. Что осталось как риск/ограничение
