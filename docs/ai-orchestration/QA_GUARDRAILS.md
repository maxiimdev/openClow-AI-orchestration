# QA_GUARDRAILS (v1)

## Цель
Снизить риск критичных багов в ЛК/регистрации за счет обязательного QA-контура перед `ready_pr`.

## Где у нас максимальный риск
1. Роли и права (переключение контекстов регистрации)
2. Персистентный стор (залипание/сброс данных между сценариями)
3. OTP/refresh/token expiry (переходы между шагами)
4. Наслаивание похожих flow (обычная vs партнерская регистрация)

## Обязательный preflight перед implement
- [ ] В тикете явно указаны роли/сценарии и отличия от смежных flow
- [ ] Явно описаны критичные инварианты (что не должно теряться/меняться)
- [ ] Зафиксированы edge-cases: refresh, timeout, back/forward, повторный вход
- [ ] Согласованы трактовки спорных API-флагов (если есть неоднозначность)

## Обязательный review/test контур
Перед `ready_pr` должны быть закрыты все 4 блока:

### A) Contract checks (front/back)
- [ ] Проверены ключевые флаги/поля API на целевых сценариях
- [ ] Негативные ветки не ломают базовый вход
- [ ] Нет неявной зависимости от одного спорного флага без fallback

### B) State integrity checks
- [ ] Роль не теряется после refresh/token cycle
- [ ] Персистентные поля не «протекают» между разными регистрациями
- [ ] Перезапуск шага не сбрасывает критичные данные без явного правила

### C) Flow regression checks
- [ ] Базовый happy-path проходит
- [ ] 2–3 критичных edge-path проходят
- [ ] Смежный flow (обычный/партнерский) не деградировал

### D) Minimal automated tests
- [ ] Добавлен хотя бы минимальный набор тестов на критичные инварианты
- [ ] Для найденного бага добавлен regression test (где это реально возможно)

## Stop rules (нельзя в ready_pr)
- Есть сомнение в сохранности ролей/критичных данных
- Нет проверки token/refresh ветки для auth-сценария
- Нет проверки смежного flow после правок общего стора
- Известный дефект без явного документированного компромисса

## Практика «не убиться об рутину»
- Делим задачу на меньшие итерации (5–7 файлов)
- После каждого цикла фиксируем, что уже проверено (чтобы не проверять хаотично)
- Ручной регресс запускать по приоритетному short-list, а не «всё подряд каждый раз»

## Обязательный deliverable в PR
1. Что изменено
2. Какие инварианты защищены
3. Какие сценарии проверены вручную
4. Какие тесты добавлены
5. Что осталось как риск/ограничение
