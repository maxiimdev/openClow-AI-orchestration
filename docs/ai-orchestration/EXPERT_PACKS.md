# EXPERT_PACKS

## Зачем нужны Expert Packs
Expert Pack — это формализованный «пакет экспертности» для конкретной роли.

Он нужен, чтобы агент:
- не был «усредненным»;
- опирался на конкретные правила и проверенные источники;
- работал предсказуемо между сессиями;
- выдавал результат, соответствующий стандартам проекта.

Ключевая идея: **экспертность = роль + правила + источники + проверка**.

---

## Что входит в Expert Pack
Для каждой роли создается отдельная папка с фиксированной структурой:

- `ROLE.md`
  - кто это за роль;
  - зона ответственности;
  - границы (что не входит в роль).

- `RULES.md`
  - обязательные правила выполнения;
  - запреты/антипаттерны;
  - приоритеты в спорных решениях.

- `SOURCES.md`
  - список источников знаний;
  - уровень доверия к источникам;
  - статус ревизии (проверено/не проверено).

- `CHECKLIST.md`
  - критерии готовности результата;
  - контрольные вопросы перед финальным ответом.

- `PROMPT.md`
  - системная инструкция для режима этой роли;
  - формат входа/выхода;
  - ограничения по объему ответа.

---

## Стартовый набор ролей (v1)
1. `frontend-lead`
2. `code-reviewer`
3. `test-engineer`

Почему именно эти три:
- они закрывают основной контур «сделать → проверить → подтвердить»;
- дают максимум пользы без раздувания системы;
- легко маппятся на режимы Claude Code.

---

## Маппинг ролей на режимы
Bridge выбирает Expert Pack по режиму:
- `implement` -> `frontend-lead`
- `review` -> `code-reviewer`
- `tests` -> `test-engineer`

Это позволяет запускать один и тот же Claude Code в разных «профессиональных режимах»,
не плодя отдельные тяжёлые агентные процессы.

## Политика качества источников
### Базовый приоритет источников
1. Документация проекта (высший приоритет)
2. Внутренние гайды команды/студии
3. Внешние материалы (статьи/книги/разборы)

### Правила включения источников
- В базу попадает только curated-контент.
- Спорные/сыррые материалы не включаются без ручной проверки.
- Каждый важный источник должен иметь пометку актуальности.

### Важный риск
Плохой источник = «уверенно неправильная» экспертность.
Поэтому качество базы важнее объема базы.

---

## Процесс обновления Expert Packs
1. Выявить проблему/пробел в текущем поведении роли.
2. Обновить `RULES.md` или `CHECKLIST.md`.
3. При необходимости добавить/удалить источники в `SOURCES.md`.
4. Зафиксировать изменение в `DECISIONS_LOG.md`.
5. Проверить влияние на качество на 2–3 реальных задачах.

Принцип: обновлять пакеты маленькими шагами, с валидацией, без резких перепрошивок.

---

## Антипаттерны (чего избегаем)
- Один эксперт «на всё сразу».
- Слишком длинные промпты без измеримой пользы.
- Накопление источников без ревизии.
- Добавление новых ролей до подтвержденной необходимости.

---

## Критерий зрелости Expert Pack
Пакет считается рабочим, если:
- роль стабильно выдает повторяемо качественный результат;
- замечания по качеству снижаются от итерации к итерации;
- токен-расход остается в рамках v1 policy;
- поведение роли понятно и объяснимо пользователю.
