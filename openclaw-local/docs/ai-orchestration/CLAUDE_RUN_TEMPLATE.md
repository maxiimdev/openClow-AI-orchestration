## Interruption Policy
- AUTO: решай сам безопасные технические детали в рамках scope.
- CONFIRM: остановись и задай вопрос, если есть 2+ равнозначных бизнес-варианта или риск затронуть соседний модуль.
- STOP: обязательно остановись при риске данных/доступов, изменении API-контрактов или выходе за scope.

## Prompting protocol (v2)
1) Сначала прочитай обязательные контекстные файлы.
2) Прочитай и примени обязательные skills для этой задачи (см. блок Skill Usage Policy).
3) Перед реализацией перечисли 3 ключевых правила, которые критичны именно для этой задачи.
4) Дай план выполнения.
5) Начинай реализацию только после явного согласования плана.

## Skill Usage Policy (mandatory)
- `required_skills`: {{REQUIRED_SKILLS}}
- `skill_usage_mode`: mandatory
- `skill_override`: запрещен без явного объяснения в отчете

Правило:
- Если required_skills заданы, ты обязан использовать их по назначению во время выполнения.
- Если какой-то skill неприменим, явно укажи причину и предложи эквивалентный подход.

## UI Verify Flag (v1)
Добавь в prompt явный флаг:
- `ui_verify_mode: auto` (default) — включать UI verify для UI-задач по протоколу.
- `ui_verify_mode: off` — отключить UI verify (только если это НЕ UI-задача или есть явное подтверждение человека).
- `ui_verify_mode: strict` — усиленная UI-проверка (доп. viewport/state checks).

Правило безопасности:
- Если выбрано `off`, исполнитель обязан в отчете указать причину и риск, а оркестратор должен зафиксировать override в task evidence.

## Fallback rule
Если не уверен — выбирай наиболее безопасный вариант, не расширяй scope, и явно опиши компромисс в отчете.
