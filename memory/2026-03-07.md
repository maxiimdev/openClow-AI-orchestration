## Critical incident: "success reports" != real public deployment

- Root issue: deployment verification was repeatedly done against the wrong runtime path/port, while public domain served a different upstream.
- Public domain `72-56-127-225.sslip.io` is routed by Caddy to `127.0.0.1:3300`.
- Several earlier restarts happened on a different process/port, which made reports look green while user still saw old build.
- Durable lesson: for user-facing deploy tasks, success criteria must include **public URL canary checks** (asset hash markers, feature markers, and HTTP checks), not just local process/build checks.

## Process lessons (durable)

- Avoid long, abstract "claude step 3/4" mega-tasks for ops incidents; they burn time/tokens and time out.
- For infra/deploy issues, use short deterministic runs with explicit canary gates and fail-fast diagnostics.
- Keep user updates proof-based and immediate when starting/running/completing tasks (task ID, phase, concrete evidence).
- Sleep mode on the worker Mac can stall claimed tasks; treat long idle claimed tasks as likely stale and requeue/clean up.
